# ShopSimulator 2.0 项目需求文档

---

## 一、版本定位

**1.0 → 2.0 核心变化**：从“能跑”到“正规化”。引入全局唯一编码体系（UID/SID/PID/OID），强化数据独立性，补全用户自助设置功能。

---

## 二、新增功能清单

### 2.1 个人设置

| 功能 | 说明 |
|------|------|
| 修改用户名 | 需检查唯一性，所有角色可用 |
| 修改密码 | 需验证旧密码，所有角色可用 |
| 访问路径 | `/settings`，所有已登录角色可访问 |

### 2.2 全局编码体系

| 编码类型 | 格式 | 生成时机 | 生成方式 | 是否可改 |
|----------|------|----------|----------|----------|
| **UID**（用户编号） | 6 位数字，如 `241507` | 用户注册时 | 随机生成，查重后写入 | ❌ 不可改 |
| **SID**（店铺编号） | 2 位大写字母 + 4 位数字，如 `IT2526` | 首次创建店铺时 | 随机生成，查重后写入 | ❌ 不可改 |
| **PID**（商品编号） | SID + `-` + 用户输入的 3 位数字，如 `IT2526-001` | 上架商品时 | 商户手动输入 3 位数字，系统拼接 SID，同一店铺内不可重复 | ❌ 不可改 |
| **OID**（订单编号） | PID-UID-数量-时间戳，如 `IT2526-001-241507-02-2026-05-08-17-45-34` | 下单时 | 系统自动拼接生成 | ❌ 不可改 |

**编码规则细节**：

- **UID 冲突处理**：随机生成 → 查 `users` 表 → 若已存在则重新生成，最多重试 10 次
- **SID 冲突处理**：随机生成 → 查 `shops` 表 → 若已存在则重新生成，最多重试 10 次
- **PID 规则**：商户上架商品时手动输入 3 位数字（001-999），系统自动在前面拼接 `<SID>-`，同一店铺内不可重复。前端可提示下一个可用编号
- **OID 格式**：`{PID}-{UID}-{quantity}-{YYYY-MM-DD-HH-mm-ss}`
  - 数量为下单时填入的值（1-99），两位数字，不足两位左补 0（如 `02`）
  - OID 创建后永久不变，数量以数据库存储值为准

### 2.3 商品详情强化

| 字段 | 类型 | 说明 |
|------|------|------|
| `alias` | VARCHAR(100)，可选 | 商品别名，如“绿水鬼” |
| `tags` | VARCHAR(255)，可选 | 商品标签，逗号分隔，如 `奢侈品,腕表,限量版` |

**使用场景**：
- 商品详情页展示别名和标签
- 后续可按标签筛选/搜索商品（本期可选实现）

### 2.4 订单表重构（去外键化）

**设计原则**：订单一旦生成，即为独立历史快照。不依赖任何外键关联，原始数据删除不影响订单记录。

**旧 orders 表字段**：

| 字段 | 处理方式 |
|------|----------|
| `id`（自增主键） | 保留 |
| `user_id`（FK → users） | **删除**，改为冗余存储 |
| `product_id`（FK → products） | **删除**，改为冗余存储 |
| `shop_id`（FK → shops） | **删除**，改为冗余存储 |
| `quantity` | 保留 |
| `total_price` | 保留 |
| `created_at` | 保留 |

**新 orders 表字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT，PK，自增 | 主键 |
| `order_oid` | VARCHAR(100)，UNIQUE | 订单业务编码 |
| `user_uid` | VARCHAR(6) | 下单用户 UID |
| `username` | VARCHAR(50) | 下单用户名（快照） |
| `shop_sid` | VARCHAR(6) | 店铺 SID（快照） |
| `shop_name` | VARCHAR(100) | 店铺名（快照） |
| `product_pid` | VARCHAR(20) | 商品 PID（快照） |
| `product_name` | VARCHAR(100) | 商品名（快照） |
| `price_at_order` | DECIMAL(12,2) | 下单时商品单价（快照） |
| `quantity` | INT | 购买数量（1-99） |
| `total_price` | DECIMAL(12,2) | 交易总额 |
| `created_at` | DATETIME | 下单时间 |

**影响分析**：

| 影响 | 处理 |
|------|------|
| 原有订单数据 | 需编写迁移脚本，将旧数据转为新格式 |
| 查询订单列表 | 不再 JOIN，直接从 orders 表读取（性能更好） |
| 删除用户/商品/店铺 | 不再受外键约束，可直接删除，订单记录保留 |
| 管理员删除订单 | 照常支持 |

## 三、数据库变更汇总

### 3.1 users 表新增字段

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `uid` | VARCHAR(6) | UNIQUE，NOT NULL | 用户唯一编号 |

### 3.2 shops 表新增字段

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `sid` | VARCHAR(6) | UNIQUE，NOT NULL | 店铺唯一编号 |

### 3.3 products 表变更

| 变更 | 字段 | 类型 | 说明 |
|------|------|------|------|
| 新增 | `pid` | VARCHAR(20)，UNIQUE，NOT NULL | 商品唯一编号 |
| 新增 | `alias` | VARCHAR(100)，可选 | 商品别名 |
| 新增 | `tags` | VARCHAR(255)，可选 | 商品标签（逗号分隔） |

### 3.4 orders 表完全重建

建表 SQL 参考：

```sql
CREATE TABLE orders (
    id              INT PRIMARY KEY AUTO_INCREMENT,
    order_oid       VARCHAR(100) UNIQUE NOT NULL,
    user_uid        VARCHAR(6) NOT NULL,
    username        VARCHAR(50) NOT NULL,
    shop_sid        VARCHAR(6) NOT NULL,
    shop_name       VARCHAR(100) NOT NULL,
    product_pid     VARCHAR(20) NOT NULL,
    product_name    VARCHAR(100) NOT NULL,
    price_at_order  DECIMAL(12,2) NOT NULL,
    quantity        INT NOT NULL DEFAULT 1,
    total_price     DECIMAL(12,2) NOT NULL,
    created_at      DATETIME DEFAULT NOW()
);
```

## 四、API 变更汇总

### 4.1 新增接口

| 方法 | 路径 | 说明 | 角色 |
|------|------|------|------|
| PUT | `/api/user/settings` | 修改用户名 `{username}`，需检查唯一性 | 所有已登录 |
| PUT | `/api/user/password` | 修改密码 `{oldPassword, newPassword}` | 所有已登录 |
| GET | `/api/user/profile` | 获取个人信息（含 UID） | 所有已登录 |

### 4.2 修改接口

| 接口 | 变更内容 |
|------|----------|
| `POST /api/auth/register` | 注册时自动生成 UID，返回 `uid` |
| `POST /api/merchant/shop` | 创建店铺时自动生成 SID |
| `POST /api/merchant/products` | 上架时接收用户输入的 3 位数字，拼接为 PID；新增 `alias`、`tags` 参数 |
| `PUT /api/merchant/products/:id` | 支持修改 `alias`、`tags` |
| `POST /api/orders` | 下单时生成 OID，订单表写入快照字段而非外键 |
| `GET /api/products` | 响应增加 `pid`、`alias`、`tags` |
| `GET /api/products/:id` | 响应增加 `pid`、`alias`、`tags` |
| 所有用户列表接口 | 响应增加 `uid` |
| 所有店铺列表接口 | 响应增加 `sid` |

### 4.3 影响范围

| 接口 | 影响 |
|------|------|
| 所有返回订单的接口 | 字段名从 `user_id/product_id/shop_id` 变为 `order_oid/user_uid/product_pid/...` |
| 管理员、商户、用户的订单查询 | 前端需要同步修改展示字段 |

## 七、风险与注意事项

| 风险 | 应对 |
|------|------|
| 旧订单数据迁移失败 | 迁移前备份数据库；迁移脚本用事务包裹，失败则回滚 |
| UID/SID 随机碰撞概率增加 | 设置重试上限（10 次），超出抛错，提示用户稍后重试 |
| 前端字段名大量变更 | 全局搜索 `user_id`、`product_id`、`shop_id`，逐一替换 |
| 删除外键后数据完整性依赖应用层 | 下单时严格校验商品存在且状态为 active，扣款前检查余额 |

## 八、2.0 完成后的项目状态

| 维度 | 1.0 | 2.0 |
|------|-----|-----|
| 用户标识 | username | username + UID（永久唯一） |
| 店铺标识 | id | id + SID（业务编码） |
| 商品标识 | id | id + PID（带店铺前缀） |
| 订单标识 | id | id + OID（可读业务编码） |
| 订单数据独立性 | 依赖外键 | 快照独立，删商品不影响订单 |
| 用户自助设置 | 无 | 改用户名/密码/查看 UID |
| 商品信息丰富度 | 名称+描述+价格 | +别名+标签 |

> **2.0 的核心哲学**：每一条数据都有独一无二的身份证，每一笔订单都是封存的历史快照。
